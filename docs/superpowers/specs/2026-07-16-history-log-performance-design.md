# 历史记录与任务日志性能改造设计

## 目标

解决历史记录和任务日志长期积累后导致的界面卡顿、任务状态更新迟缓和批量下载吞吐下降，同时完整保留现有历史记录。

## 历史记录存储

将 `%LOCALAPPDATA%/geo-downloader/history.json` 迁移到 SQLite 数据库 `history.db`。

历史表保存现有 `DownloadRecord` 全部字段，并增加日志元数据：

- `log_path`
- `log_compressed`
- `log_original_size`
- `log_stored_size`
- `log_truncated`
- `log_updated_at`

按 `created_at DESC` 建索引。新增、删除、更新历史均使用单条 SQL，不再读取和重写全部记录。所有 SQLite 操作从异步命令中通过 `spawn_blocking` 执行，避免阻塞下载运行时。

## 旧数据迁移

首次打开数据库时检测 `history.json`：

1. 创建数据库和表结构。
2. 在单个事务中批量导入 JSON 中的全部记录。
3. 使用记录 ID 作为主键，重复启动时通过冲突忽略保证幂等。
4. 事务提交成功后将原文件重命名为 `history.json.migrated`。
5. 迁移失败时回滚事务并保留原文件，向前端返回明确错误，不静默清空数据。
6. 损坏 JSON 不自动覆盖，保留原文件并报告迁移失败。

迁移在首次历史访问时执行，不阻塞应用启动和下载任务启动。

## 分页接口与前端

历史查询接口接受 `page` 和 `page_size`，返回：

```text
records, total, page, page_size
```

默认每页 50 条，后端限制每页最大 200 条。前端仅渲染当前页，显示总记录数、当前页和总页数，并提供上一页、下一页。

历史面板只监听 `download-history-updated`。移除通过 `task-list-updated` 触发的历史全量刷新。

删除当前页最后一条记录后，如果当前页已超出新的总页数，自动回到最后一个有效页面。

## 日志存储

日志正文继续保存在独立文件中，不写入 SQLite。SQLite 只保存路径及压缩、大小、截断状态。

原因：

- 日志是追加写，文本文件成本更低。
- 避免高频 SQL 写入和数据库 WAL 膨胀。
- 历史数据库损坏时不同时丢失诊断日志。
- 日志仍可直接查看和导出。

## 日志写入隔离与大小限制

磁盘日志写入不得在 `TaskManager` 全局任务锁内执行。任务条目中的日志文件句柄使用独立锁，内存日志状态和文件写入分离，避免慢磁盘或实时杀毒扫描阻塞其他任务状态更新。

每个任务日志原始内容上限为 5 MiB。达到上限后：

- 不再追加普通日志；
- 写入一次明确的“日志已达到 5 MiB，后续内容已截断”记录；
- 将历史元数据标记为 `log_truncated`。

## 压缩和保留策略

- 任务完成后 7 天内：保留 `.log`。
- 超过 7 天：压缩为 `.log.gz`，压缩成功后删除原 `.log`。
- 超过 90 天：删除 `.log` 或 `.log.gz`，历史记录继续永久保留，并清空数据库中的日志路径和大小元数据。
- 删除单条历史：同时删除关联日志文件。
- 清空历史：删除所有历史记录及其关联日志，但不得删除活动任务日志。
- 启动维护：清理数据库中不存在关联记录的孤儿日志，但跳过当前活动任务日志。

压缩使用 gzip。历史日志查看接口透明支持 `.log` 和 `.log.gz`，前端操作不变。

## 后台维护

日志压缩和清理由低优先级后台线程执行：

- 不阻塞应用启动；
- 不持有任务管理器全局锁；
- 每轮限制处理数量，避免首次升级时集中处理大量文件占满磁盘；
- 单文件压缩使用临时文件，成功后原子重命名，再删除原日志；
- 失败时保留原日志并记录警告，下次维护重试。

首次升级不立即压缩所有旧文件。后台按批次逐步处理，下载优先。

## 并发与一致性

历史数据库使用 SQLite WAL 模式和短事务。写操作由共享的历史存储实例协调，避免当前每次 `HistoryManager::new()` 创建独立无效锁的问题。

历史记录写入完成后再发送 `download-history-updated`。日志维护更新元数据时不触发整页刷新，除非当前记录的日志可用状态发生变化。

## 错误处理

- 历史写入失败不能把已经成功的下载改为失败，但必须写应用日志并向历史面板显示持久化警告。
- 日志写入、压缩或清理失败不得影响下载结果。
- 读取已过期或丢失的日志时返回“日志已清理或不存在”，不返回空白成功。
- 数据库初始化失败时保留旧 JSON，不执行破坏性迁移。

## 测试

后端测试覆盖：

- 完整 JSON 迁移及字段保持；
- 迁移重复执行的幂等性；
- 损坏 JSON 和事务回滚；
- 分页顺序、总数和边界参数；
- 单条新增、删除、清空、金字塔状态更新；
- 并发新增不丢记录；
- 5 MiB 日志截断；
- 7 天压缩、90 天删除；
- gzip 日志透明读取；
- 删除历史同步删除日志；
- 活动日志不被维护任务删除；
- 孤儿日志分批清理。

前端验证覆盖：

- 4 万条历史时仅请求和渲染当前 50 条；
- 翻页、刷新、删除和清空行为；
- 新任务完成后刷新当前页及总数；
- 压缩日志和普通日志均可正常查看；
- 历史面板打开时批量下载仍能并行推进。
