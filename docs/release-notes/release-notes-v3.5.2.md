# v3.5.2 — 下载过程可见性 hotfix + 启动清孤儿临时目录

针对群友反馈的"下载很久但缓存 db 没增长、临时目录是空的"和"打开/关闭主窗体老是弹未保存更改"两组问题做的 hotfix 修复。

## 关键修复

- **设置面板不再误弹"未保存更改"弹框**
  react-hook-form 的 isDirty 在初始 reset 阶段短暂为 true，配合 React 18 StrictMode 双挂载，触发了关窗确认弹框。主进程已经统一处理关窗（最小化到托盘），前端无需重复拦截，移除即可。

- **下载过程实时显示无数据瓦片数**
  之前 HTTP 404 的瓦片被静默归为 no_data，进度涨到 80% 但临时目录是空的、缓存 db 没增长，用户完全无感知。现在：
  - 任务面板 message 实时显示 `下载中 X/Y，无数据 N`
  - 10% 进度日志同步附加无数据张数
  - 大量 404 时（≥100 张且占已完成 ≥ 50%）单独推送一条 WARN 日志：`已有 N 张瓦片返回 404（占 X%），图源在此区域/级别可能无覆盖，建议降低缩放级别后重试`

- **缓存写入失败不再静默吞掉**
  原来 `let _ = tile_cache.put(...)` 忽略所有错误，磁盘满 / SQLite busy / 权限问题完全摸瞎。现在显式 `log::warn!` + 通过原子计数器累计，累计 ≥ 50 次失败时主动推送一条 WARN 日志告知用户。

- **启动时自动清理 >24h 的孤儿临时目录**
  之前进程被强杀 / 上次运行残留会留下 `%TEMP%/tif-dl-*` 目录，时间久了越积越多。现在启动时异步扫描清理 mtime > 24h 的目录（保留 24h 内的，避免误删 CompletedWithGaps 任务待用户决策的 temp）。

## 与"自动导出最低成功率"机制的关系

三层告警**完全互补**，文案不冲突：

| 触发时机 | 文案 | 决策点 |
|---|---|---|
| 下载中实时（≥100 张 + ≥50% no_data） | 已有 N 张瓦片返回 404（占 X%）... | 用户可早取消 |
| 缓存写入累计 ≥ 50 次失败 | 缓存写入失败累计 N 次... | 用户排查磁盘/权限 |
| 下载完成后 success_ratio < 阈值 | z18 成功率 X% 低于阈值 Y%，跳过导出 | 用户补漏/强制导出 |

## 内部清理

- 修正 `ZoomLevelResult.success_count` 字段注释与阈值判断处的注释，与实际行为（no_data 计入 failed_count 分母失败侧、阈值能拦截全 404 任务）保持一致。

## 不影响

下载主流程 / 缓存命中逻辑 / 重试策略 / cleanup_temp_dir / 任务取消暂停失败分支 / Issue #32 raw tiles 直写 / Issue #31 阈值机制。
